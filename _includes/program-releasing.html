Working in the open

Teams at Microsoft are encouraged to work in the open and are provided with straightforward guidance on when and how to release code to the world.

Release policy

This is a summarized, distilled look at general open source release requirements at Microsoft. Microsoft employees can find the complete guidance at aka.ms/opensource.

While releasing Microsoft code as open source is encouraged, we also need to protect valuable Intellectual Property and ensure a valuable and quality experience for users of our projects.

Approval process

All open sourcing of Microsoft source code and content (e.g., text, images, fonts, data) must be registered with your business and legal team using the workflow outlined in the release documentation.

In many cases, this process is lightweight and instant.

In other cases, you may need to work with your management and legal teams to review the business case, success plan, and IP management strategy around your proposal.

If you are looking to release non-product code (such as utility libraries, tools, and sample code), your registration will not require further review if a set of specific conditions are met around good judgement, publishing with the MIT license, not distributing third-party code or content, not containing telemetry, not containing cryptography, and other specific requirements that have been listed and made available internally to Microsoft employees for review.

As an example: a small utility library for managing configuration, released with the MIT license, may not require a review.

Another example: an artificial intelligence algorithm intended to be released under the GPL license does require review.

Location

All Microsoft open source code should be released on GitHub in one of Microsoft's GitHub Enterprise Cloud orgs. The vast majority of projects will go in the official Microsoft org.

Where we are participating in existing community that does not work on GitHub, your code should go in the natural location for that community.

If you do not plan on releasing your code to the public, an InnerSource location provided by the One Engineering System team is a better option for private engineering work.

The following projects are typically hosted in a private engineering system and are not permitted on GitHub.com for open source:

  • First-party product code
  • Projects only shared with the rest of the company (new hire wiki guides, internal hackathon projects, for example)
  • Moonlighting projects
Security

All Microsoft open source code must follow all applicable security requirements related to the Security Development Lifecycle.

Licensing

All Microsoft open source code should be released under the MIT license.

Where we are participating in existing community that does not use the MIT license, work with your business and legal contacts to determine the appropriate license. Our intention is to fit in with the existing community. We just need to confirm that the terms work for the business.

Contributions

All contributions to a Microsoft-managed open source project must be covered by a Contributor License Agreement (CLA) that validates that the contributor(s) have the right to contribute the content and that Microsoft has the right to use, distribute and relicense the contribution.

The CLA must be agreed to by all contributors who are not Microsoft Full-time Employees (FTE) or interns prior to the contribution being merged into the project codebase.

The CLA requirement is waived for contributions that only affect one file and with a change that is 15 lines or less, or whatever the current internal guidance is for this policy.

Release checklist

Releasing an open source code project is a different practice than releasing a product or service. This content only refers to standing up an open source project to a public repository on GitHub.

The following steps should be taken to release and maintain your open source project using the MIT license:

  • Register your release: You can start your release process anytime using the internal source portal, or by creating your repo on GitHub directly and following the instructions you receive by e-mail. If a business and legal review is needed, the wizard process will trigger that easy review process.
  • Create and initialize a GitHub repository: create your GitHub repository using the new repository wizard or directly on GitHub. Repositories must be hosted within a Microsoft-managed GitHub Enterprise Cloud organization and may not be made public until the release registration is approved by your legal team and management, as required.
  • Include the following in the root directory:
    • A README.md file describing the purpose and state of the repository, with a link to the Microsoft Open Source Code of Conduct and a description of any third-party code included in the repository. A good README file is often critical to project success.
    • A LICENSE.txt file with the MIT license text. Licenses other than MIT must be cleared with your open source attorney team.
    • A CONTRIBUTING.md file with instructions on contributing to the project that must include at least the minimum required information. That minimum information includes that contributions to Microsoft projects are subject to our Contributor License Agreement. Additionally, consider providing technical guidance like build instructions, coding conventions, or a project roadmap in the CONTRIBUTING.md file.
    • Telemetry Notice if the release has code that enables Microsoft to collect telemetry, also include in the repository instructions to turn off the telemetry and a specific data collection notice in a the README.md file.
    • Trademark Notice your README should include the standard trademark notice that is a part of the default new repo templates.
    • Security Reporting Instructions: Include instructions that enable users to privately report security vulnerabilities found in your project. Sample instructions are available inside the company internal docs.
  • Prepare the code for release:
    • Code license: The only pre-approved license is the MIT license - other licenses must be cleared with your legal team. To apply the MIT license to your code, include the language-appropriate comment at the head of each source file.
    • Remove sensitive assets: XXX
      • Remove any reference in the code to internal or confidential information, including internal paths, tools, codenames, proprietary fonts, internal telemetry and email aliases.
      • Remove any trademarks or product icons.
      • Prepare the code and commit history for publication, including running certain internal corporate tools for this purpose.
    • Third-party Open Source: If your repository includes third-party OSS, describe its use and its license in a NOTICE file. When in doubt, consult with your open source attorney team.
  • Publish the code: Once your registration and review process is complete, make your GitHub repository public.
  • Going forward: Ensure:
    • Further Microsoft Contributions: You must file a contribution request if either the feature you are adding is outside the scope of your original release request OR you are adding non-open source Microsoft code that your team did not author.
    • Staffing: Ensure at least one team member is committed to managing community interactions merging pull requests, giving feedback, releasing new versions.
    • Buildable & Runnable: Make sure that your project is easy to build and has binaries available is key to attracting contributors.
    • Foster your community: Look for ways to continue engagement with potential contributors.
